# NotebookLM Packaging Rules

Use these rules to prepare handover material for NotebookLM.

## Objective
NotebookLM should receive clean, bounded, low-duplication source documents that can stand on their own.

## Classification buckets

### Upload as-is
Use when the document is already:
- clearly titled
- scoped to one project/process/topic
- written in complete sentences or structured bullets
- low on duplication
- understandable without the original chat

Examples:
- project handover page
- SOP
- decision log
- glossary
- architecture note

### Clean before upload
Use when the content is valuable but still noisy.

Typical cleanup actions:
- add title
- add section headings
- merge duplicates
- remove brainstorming debris
- separate final content from draft content
- normalize terminology
- add context paragraph at top

Examples:
- mixed raw notes
- chat-derived summary with no headings
- partial project page
- unstructured research summary

### Do not upload
Use when the source adds more confusion than value.

Common cases:
- superseded drafts
- repetitive copies of the same document
- scratch notes with no decisions
- fragmented chat excerpts with no context
- files that are mostly operational noise

## Packaging recommendations
1. One source = one clear topic
2. Prefer 1–5 page source docs over giant mixed dossiers
3. Create a separate glossary if terminology is inconsistent
4. Separate project pages from raw source material
5. Maintain a top-level index outside or inside NotebookLM for navigation

## Minimal upload pack
If time is very limited, upload:
- Master Index
- project pages for key workstreams
- Decision Log
- Open Issues
- Glossary

## Anti-patterns
Avoid:
- uploading entire chat histories without curation
- mixing multiple projects in one source
- uploading unresolved brainstorms as if final
- storing duplicate summaries with different wording
